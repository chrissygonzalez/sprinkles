Class Cards

Class Cards is my submission for the Google Interaction Design exercise. I chose the Names and Faces prompt, which asks for the designer to “design an experience to help an educator match faces to names, with the goal of shortening the time needed to reach complete un-aided accuracy.”

Teacher Types

I started work by thinking about teachers and where they teach. There’s a lot of variation in schools and classroom types, but at a high level, one of the main differences in context is whether a teacher stays with one class all day or whether they teach multiple classes in periods.

A single class has a limited number of names and faces to remember, but that can expand quickly with multiple classes. How many students might be in a class, and how many classes might be in a day? I did a quick search and found that 30 students is an average class size, and school days can have up to eight periods. So a teacher with multiple classes might need to learn up to 240 names and faces, in chunks of 30. I chose to focus on teachers with multiple classes, with the assumption that my solution could also scale down to meet the needs of a teacher with a single class.

Learning Names

Next, I wanted to learn more about how to memorize peoples’ names (I’m not great at this myself). Teachers have been learning their students’ names and faces for a long time, so I thought they might be a good source for tips on doing it quickly. I found plenty of articles and blog posts on the topic, from which I gathered these seven tips:

  1. Practice saying names out loud repeatedly.
  2. Relate faces to interests.
  3. Think of funny, rhyming, or alliterative word associations.
  4. Memorize in smaller groupings.
  5. Hear how the student says their name.
  6. Write, dance, or draw names and faces.
  7. Have students sit in the same seats at first.

Flow and Sketches

Once I knew something about learning names and the number of them I needed to accommodate, I started to think about how teachers could get students’ photos. Ideally, students would send them in before school started so teachers could practice them early, so I spent some time thinking about how that flow might work.

I thought the student experience would be more fun (and get more submissions) if it allowed for personal expression. Then teachers could use the student-chosen interests, visualizations, and alliterative adjectives to learn faster.

High Fidelity

I chose the main teacher use case – having existing classes and studying their names – for high-fidelity mocks. My goal was to make the site feel fun but clear, with space for many classes or just a few. Having seen a few public schools in New York, I was worried that students (and maybe teachers) might not have their own phone or computer. I designed this as a website rather than an app to make it easier to use a borrowed or public device. Clickable prototype

Mobile Web

I wanted teachers to be able to study student names from any device, so I made sure the core screens would work on mobile web. I would also want the student photo survey to work well on mobile. Clickable prototype

Next Steps

If this was a real product, I would want to talk to some teachers and students at this point and consider what might change if it’s used internationally. I made a lot of assumptions in this design process, the main one being that teachers would actually want to learn and manage another classroom tool. It would be interesting to know what other services teachers might need this to integrate with, and whether putting so much student data into a new system could pose a privacy risk. Despite my concerns, I had a great time with this prompt, and I think I may try some of these techniques to get better with names myself.
